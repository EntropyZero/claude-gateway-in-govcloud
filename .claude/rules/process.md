# Rules — working process

- **All tests pass before moving to the next task.** Run `make test` (or at
  least the affected suite) and get it **green** before starting the next unit
  of work or committing. A red suite blocks progress — fix it or revert; never
  stack new work on a failing base. New behavior ships with a test in the
  matching suite (`tests/lambda` pytest, `tests/bash` bats, `tests/cfn`
  cfn-lint+cfn-guard, `tests/powershell` Pester). A gate that can't fail is not
  a test — when adding one, confirm it goes red on a real violation.

- **Self-review before committing nontrivial infra changes.** For any change
  to CloudFormation, deploy scripts, or the db-admin Lambda, run a multi-agent
  review of the diff before committing: independent finder angles (line-by-line,
  removed-behavior, cross-file) **plus** an adversarial pass that web-checks the
  AWS / Postgres / Grafana semantics the change assumes. On this repo that pass
  has repeatedly caught deploy-breakers that syntax checks and docs missed
  (health-check protocol default, endpoint-policy gaps, installer TOCTOU, a
  guard that silently disabled telemetry). Skip it only for docs/diagram-only
  or trivial mechanical edits.

- **Verify locally, every time:** `bash -n` each changed script, YAML-parse the
  changed templates, `py_compile` the Lambda. These are cheap; run them.

- **"Done" means verified live.** Doc- or web-confirmed behavior is a
  hypothesis until a real deploy exercises it. Flag anything that is
  assumption-verified-only (the async rotation event shape is the standing
  example) as needing test-run confirmation, and don't mark it settled sooner.

- **Documentation visuals: SVG or PNG only — NEVER Mermaid.** This applies
  to every visual in any documentation (architecture diagrams, README
  figures, runbook illustrations, review packages), not just the diagram
  set. Mermaid's auto-layout produced unreadable and partly non-rendering
  output here, and its rendering depends on the viewer — an SVG/PNG looks
  the same everywhere. The architecture diagrams specifically are
  hand-laid-out SVGs generated by `docs/diagrams/generate.py`: edit the
  script, regenerate, **rasterize and actually look at the output**
  (cairosvg) before committing, and commit both script and SVGs. For any
  new visual elsewhere, follow the same pattern (generated or hand-authored
  SVG preferred; PNG acceptable) and eyeball the rendered result before
  committing.

- **Keep the review doc honest.** `docs/security-review-2026-07.md` is the
  source of truth for finding status and accepted risks; update it in the same
  change that alters a finding. Keep the Status block in `CLAUDE.md` current.

- **Surface accepted risks, don't bury them.** The plaintext OTLP hop (C2) and
  deferred Object Lock (C9) are deliberate, SSP-scoped decisions. Any review
  package or summary states them up front rather than letting a reviewer find
  them.

- **Commit trailer:** end messages with
  `Co-Authored-By: Claude Fable 5 <noreply@anthropic.com>`.
